## 2. 預計實作的功能與架構

### 一、核心系統架構 (System Architecture)

本系統採 Client-Server 架構，將前端展示與後端邏輯分離，確保資料安全性與操作流暢度。

* **前端 (Frontend)**:
    * 使用 HTML5, CSS3 進行頁面切版與 RWD (Responsive Web Design) 響應式設計。
    * 使用 JavaScript (Canvas API) 處理四款遊戲的核心運作邏輯與 DOM 操作。
* **後端 (Backend)**:
    * 使用 Python Flask 框架。
    * 負責處理前端發送的請求 (Request)、會員驗證邏輯 (Session 管理) 以及 API 路由控制。
* **資料庫 (Database)**:
    * 使用 SQLite 關聯式資料庫。
    * 設計資料表以儲存使用者帳號資訊 (users) 與各項遊戲分數紀錄 (scores)。

### 二、前後端與資料庫互動模式 (Interaction Flow)

資料互動主要透過 HTTP Request (GET/POST) 進行，流程如下：

1.  **會員驗證流 (Auth Flow)**:
    * 使用者在前端表單輸入帳密 -> 後端接收 POST 請求 -> 查詢資料庫 (SELECT) 比對帳密。
    * 若比對成功：後端建立 Session 並回傳登入成功訊息，前端跳轉至大廳。
    * 若失敗/格式錯誤：後端回傳錯誤代碼，前端顯示警告視窗 (Alert)。
2.  **遊戲分數上傳 (Score Submission)**:
    * 當玩家遊戲結束 (Game Over) 時，前端 JavaScript 自動抓取最終分數。
    * 透過 AJAX/Fetch 發送異步請求將「分數」與「使用者 ID」傳送至後端。
    * 後端接收後執行寫入資料庫 (INSERT) 動作，更新該玩家的歷史紀錄。
3.  **排行榜資料獲取 (Data Retrieval)**:
    * 當使用者進入排行榜頁面，前端發送請求。
    * 後端對資料庫執行排序查詢 (ORDER BY score DESC)，撈取前 10 筆資料。
    * 後端將資料包裝成 JSON 或 HTML 格式回傳，前端渲染成表格顯示。

### 三、頁面跳轉關係 (Page Routing)

本系統規劃以下頁面路由，並設有權限檢查機制 (未登入者強制導回首頁並跳出警告)：

1.  **公開層 (Public Zone)**:
    * **首頁/登入頁 (Login)**: 系統入口，提供登入與註冊按鈕。
    * **註冊頁 (Register)**: 填寫資料，成功後跳轉回登入頁。
2.  **會員層 (Member Zone - 需登入狀態)**:
    * **遊戲大廳 (Lobby)**: 登入後的中心頁面，顯示歡迎訊息，可點擊不同卡片跳轉至四款不同的遊戲頁面。
    * **遊戲頁面 (Game Room)**:
        * 包含四個獨立 URL (如 `/game/snake`, `/game/dino` 等)。
        * 遊戲結束後提供「再來一局 (Reload)」或「返回大廳 (Back to Lobby)」按鈕。
    * **排行榜 (Leaderboard)**: 可從大廳進入，上方有標籤 (Tab) 可切換查看四種不同遊戲的排名，並可跳轉回大廳。
